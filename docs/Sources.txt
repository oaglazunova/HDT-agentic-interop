In the **Interoperable-and-modular-HDT-system-prototype** repository, those three “domains” are sourced as follows.

## 1) Trivia data

**Source:** GameBus API (live, per user)

**Where in code:** `hdt_core_infrastructure/GAMEBUS_DIABETES_fetch.py` → `fetch_trivia_data(...)`

**How it is retrieved:**

* The code calls the GameBus “activities” endpoint:

  * `https://api3-new.gamebus.eu/v2/players/{player_id}/activities?gds=ANSWER_TRIVIA_DIABETES`
* It authenticates using a **Bearer token** (`auth_bearer`) taken from `config/users.json` for the user’s connected diabetes app (when it is `GameBus`).

So “Trivia” here means **GameBus activity records** under the GDS descriptor `ANSWER_TRIVIA_DIABETES`.

## 2) SugarVita data

**Source:** GameBus API (live, per user)

**Where in code:** `hdt_core_infrastructure/GAMEBUS_DIABETES_fetch.py` → `fetch_sugarvita_data(...)`

**How it is retrieved:**

* The code makes **two** GameBus activity calls (two descriptors):

  * `gds=SUGARVITA_PLAYTHROUGH`
  * `gds=SUGARVITA_ENGAGEMENT_LOG_1`
* Endpoints:

  * `https://api3-new.gamebus.eu/v2/players/{player_id}/activities?gds=SUGARVITA_PLAYTHROUGH`
  * `https://api3-new.gamebus.eu/v2/players/{player_id}/activities?gds=SUGARVITA_ENGAGEMENT_LOG_1`
* Same auth mechanism: **Bearer token** from `config/users.json`.

So “SugarVita” here means **GameBus activity records** under those two SugarVita-related GDS descriptors.

## 3) Diabetes data

**Source:** Derived locally by the model (not pulled from an external “diabetes API”)

In this repo, “diabetes data” (as exposed to apps) is primarily **computed outputs**:

* a **diabetes health literacy score**
* **SugarVita player type labels/scores**

**Where it is produced:**

* `Virtual_Twin_Models/HDT_DIABETES_model.py`

  * Fetches trivia + SugarVita via the local HDT API endpoints:

    * `http://localhost:5000/get_trivia_data`
    * `http://localhost:5000/get_sugarvita_data`
  * Computes scores using `Virtual_Twin_Models/HDT_DIABETES_calculations.py`
  * Writes results to the local file: `diabetes_pt_hl_storage.json`

**Where it is served from:**

* `hdt_core_infrastructure/HDT_API.py`

  * `/get_health_literacy_diabetes` reads from `diabetes_pt_hl_storage.json`
  * `/get_sugarvita_player_types` reads from `diabetes_pt_hl_storage.json`

Pragmatically: in this prototype, “diabetes” is **a local, persisted model result store** (`diabetes_pt_hl_storage.json`). The repo even ships with a pre-filled example file, but it is intended to be updated by running `HDT_DIABETES_model.py`.
